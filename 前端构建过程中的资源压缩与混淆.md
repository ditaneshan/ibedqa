**使用 Ansible 实现基础设施即代码（IaC）**

在团队协作中，服务器初始化、配置漂移和重复部署往往是最耗时的问题。Ansible 的价值在于把“手工操作”转化为“可版本管理的代码”，让基础设施具备可追踪、可回滚、可复用的能力。相比脚本式运维，Ansible 更强调幂等性：同一份 Playbook 执行多次，结果应保持一致，这正是 IaC 的核心。

一个常见场景是：新机器上线后，需要安装 Nginx、下发配置并确保服务启动。用 Ansible 可以把这些步骤固化为声明式任务：

```yaml
---
- name: Configure web server
  hosts: web
  become: true
  vars:
    nginx_port: 80

  tasks:
    - name: Install Nginx
      ansible.builtin.yum:
        name: nginx
        state: present

    - name: Deploy nginx config
      ansible.builtin.template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      notify: Restart Nginx

    - name: Ensure Nginx is running
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: true

  handlers:
    - name: Restart Nginx
      ansible.builtin.service:
        name: nginx
        state: restarted
```

这段代码解决了三个问题：第一，环境差异被消除，所有主机执行同一套标准；第二，配置变更可通过 `template` 和 `notify` 精准触发重启，避免无意义中断；第三，Playbook 本身进入 Git 后，配置修改就变成了可审查、可追溯的代码变更。

实践中建议关注以下几点：一是尽量使用模块而不是 `shell`，这样更稳定，也更符合幂等性；二是按职责拆分 `role`，把安装、配置、监控解耦，提升复用率；三是敏感信息使用 Ansible Vault 管理，避免明文泄露；四是在 CI 中加入 `--check` 和 `--diff`，提前发现配置偏差；五是对不同环境使用 `inventory` 和变量分层，减少硬编码。

从工程角度看，Ansible 不只是自动化工具，更是基础设施治理方式的升级。它把运维经验沉淀为代码，让“环境一致性”从口号变成可执行的标准。对于追求稳定交付的团队来说，Ansible 是落地 IaC 的高性价比选择。
